# Architecture Document: Game Table Companion

## 1. Architecture Overview

**Game Table Companion** is a single-tier, client-only mobile application with a modular monolith structure. All computation and storage are local to the device. There is no server, no network communication, and no cloud synchronization.

The architecture comprises three functionally independent feature modules (Dice, Score, Draw) sharing a common persistence layer, navigation system, and React Native UI runtime. Each module is self-contained, with clear internal organization and minimal cross-module coupling (only through navigation and a shared game context for the Score module).

**Why this approach:**
- Simple scope: all logic is local, deterministic, and contained
- Fast iteration: no server-side code to maintain or deploy
- Zero operational complexity: nothing to monitor or scale
- Offline-first by design: no network fallback required

## 2. Architecture Decision Records

### ADR-001: React Native + Expo as Primary Framework

**Decision:** Use React Native (v0.72+) managed via Expo (SDK 49+) for cross-platform iOS and Android development.

**Context:**
- Spec requires iOS 13+ and Android 8+ support simultaneously
- Team efficiency demands single codebase, not parallel native development
- Animations must run at 60 fps with native smoothness
- Rapid iteration needed for MVP in early 2026

**Alternatives Considered:**
1. **Flutter with Dart** – Excellent performance, but introduces new language, longer team ramp-up
2. **Kotlin Multiplatform** – Code sharing, but Android-first approach; iOS secondary
3. **Native (Swift + Kotlin)** – Best performance, but 100% code duplication, 2x maintenance burden
4. **Web wrapper (React + Capacitor)** – Simplest single language, but poor offline support for file I/O patterns, touch performance not guaranteed

**Trade-offs Accepted:**
- JavaScript bundle slightly larger than native (mitigated by Expo tree-shaking)
- Some APIs require native modules (haptics, file I/O), managed via Expo
- Performance ceiling is lower than pure native, but 60 fps target is achievable

**Status:** Accepted

---

### ADR-002: AsyncStorage for Game Persistence

**Decision:** Use `@react-native-async-storage/async-storage` (v1.20+) as the persistent storage engine for game sessions.

**Context:**
- Game sessions are small (< 10 KB each) and JSON-serializable
- No complex queries, filtering, or indexing needed
- No relational integrity constraints between tables
- Rapid development with zero schema design overhead

**Alternatives Considered:**
1. **SQLite (via expo-sqlite)** – Structured, indexed, queryable; overkill for this data model
2. **Firebase Realtime Database** – Cloud-hosted, violates offline-first requirement; introduces auth, privacy complexity
3. **Realm** – Powerful but heavyweight; unnecessary for simple key-value use case
4. **AsyncStorage + file system (expo-file-system)** – AsyncStorage sufficient; files only if large binary data needed (not present here)

**Trade-offs Accepted:**
- Hard limit of ~10 MB per app (acceptable for 1000+ small game sessions)
- No query API (iterate in-memory or load all; acceptable given small dataset)
- No automatic indexing on timestamp (implement in-app sorting)

**Status:** Accepted

---

### ADR-003: React Native Reanimated 2 for High-Performance Animations

**Decision:** Use `react-native-reanimated` (v3.x) as the animation engine for all UI transitions and game animations.

**Context:**
- Spec mandates 60 fps animations for dice rolling and draw results
- Animations must maintain frame rate even during simultaneous multi-touch input
- Need worklet-based animation syntax for direct thread communication (avoid bridge bottleneck)

**Alternatives Considered:**
1. **React Native Animated API** – Built-in, but runs on JS thread; potential 60 fps drops under load
2. **Lottie** – Vector animation library; overkill for procedural dice/draw effects
3. **Three.js / Babylon.js** – 3D rendering; excessive complexity for 2D UI effects

**Trade-offs Accepted:**
- Added bundle size (~150 KB gzipped)
- Worklet syntax unfamiliar to some JS developers (mitigated by clear documentation)
- Tight coupling to Reanimated API versions (stable within v3 series)

**Status:** Accepted

---

### ADR-004: React Native Gesture Handler for Multi-Touch Detection

**Decision:** Use `react-native-gesture-handler` (v2.x) for robust multi-touch input in the Draw module.

**Context:**
- Spec requires reliable detection of 10+ simultaneous touch points
- Native `PanResponder` has known limitations with concurrent touches at edge-to-edge
- Draw feature is performance-critical; needs proven stability

**Alternatives Considered:**
1. **React Native PanResponder** – Lower-level, simpler, but less robust with 5+ simultaneous touches
2. **Native Touch APIs (Objective-C/Kotlin)** – Most reliable, but bridges to JS add latency
3. **Custom GestureRecognizer wrapper** – Overkill; Gesture Handler provides proven solution

**Trade-offs Accepted:**
- External dependency (though maintained by React Native community)
- Integrates only with Reanimated 2+ (already selected)

**Status:** Accepted

---

### ADR-005: Context API + Hooks for State Management (No Redux)

**Decision:** Use React Context API with TypeScript and custom hooks; avoid Redux, MobX, or Zustand.

**Context:**
- Three independent feature modules with minimal cross-module state sharing
- Only the Score module requires persistence; Dice and Draw are stateless
- State tree is shallow: games array + current game ID + current player selections
- Simple unidirectional data flow within each module

**Alternatives Considered:**
1. **Redux** – Powerful middleware, time-travel debugging, but adds boilerplate for this scope
2. **MobX** – Reactive, but unfamiliar to many React developers; less ecosystem tooling
3. **Zustand** – Lightweight alternative, good for this scale; considered but Context API sufficient
4. **Jotai / Recoil** – Atomic state, but team preference for Context clarity

**Trade-offs Accepted:**
- Less explicit action creators; rely on naming conventions for clarity
- Prop drilling if future complexity grows (refactor to Zustand at that point)
- No built-in devtools for state inspection (use React DevTools instead)

**Status:** Accepted

---

### ADR-006: Modular Monolith Architecture

**Decision:** Organize code into feature modules (Dice, Score, Draw) sharing common services (Persistence, Navigation), rather than a flat monolithic or microservices structure.

**Context:**
- Three distinct user journeys (Dice, Score, Draw) with different data models
- Share only navigation, theme, and persistence infrastructure
- Single deployment unit (one app binary for iOS and Android)

**Alternatives Considered:**
1. **Flat Monolith** – All code at root level; harder to navigate as app grows
2. **Microservices (on-device)** – Unnecessary complexity; no independent deployment needed
3. **Component Library** – Too fine-grained; modules are not UI component libraries

**Trade-offs Accepted:**
- Shared services must have stable interfaces (any breaking change requires coordination)
- Module boundaries enforced by convention, not enforced by tooling
- Cross-module refactoring requires care

**Status:** Accepted

---

### ADR-007: No Backend or Cloud Services (Offline-First Design)

**Decision:** Eliminate any reliance on cloud backends, APIs, or services. All data stays on-device.

**Context:**
- Spec explicitly states: "designed to function completely offline, without requiring registration, login or connection to server backend"
- Users expect zero friction (no login, email validation, account setup)
- Simplifies regulatory compliance (no GDPR data processing, no third-party data sharing)

**Alternatives Considered:**
1. **Firebase Realtime Database** – Would enable future sync, but violates offline-first
2. **Custom REST backend** – Would support leaderboards, cloud backup, but massive scope increase
3. **Hybrid approach** – Try to sync if online, local if offline; adds complexity for marginal benefit

**Trade-offs Accepted:**
- Cannot support cross-device sync in future without major rearchitecture (post-v1)
- No cloud backup; users responsible for device backup (iOS iCloud, Android Google Drive)
- No leaderboards or social features (versioning dependent on v2+ scope)

**Status:** Accepted

---

### ADR-008: Bottom Tab Navigation for Three Modules

**Decision:** Implement three-tab bottom navigation (Dice | Score | Draw) using `@react-navigation/bottom-tabs`.

**Context:**
- Three independent modules of roughly equal prominence
- Mobile UX pattern: bottom tabs are faster to tap than drawer or hamburger menu
- Spec implies equal discoverability of all three features

**Alternatives Considered:**
1. **Side drawer menu** – Takes more screen space, slower to access
2. **Top tab navigator** – Competes with status bar, less thumb-friendly
3. **Hierarchical with home screen** – Adds unnecessary layer; users want direct access

**Trade-offs Accepted:**
- Bottom tab area unavailable for content (design with 20% margin at bottom)
- Tab labels must be concise (2-3 words max)

**Status:** Accepted

---

### ADR-009: TypeScript for Type Safety (Not Plain JavaScript)

**Decision:** Use TypeScript 5.0+ across all source code; strict mode enabled.

**Context:**
- Spec defines precise data models (GameSession, Player, etc.)
- Reduces runtime errors in persistence layer (data corruption risk)
- Enables IDE autocomplete for navigation params and props
- Team experience with TypeScript

**Alternatives Considered:**
1. **Plain JavaScript** – Faster initial dev, but poor refactoring safety
2. **Flow** – Type checker without compilation step, less mature than TS

**Trade-offs Accepted:**
- Build time slightly longer due to TS compilation
- Steeper learning curve for JS-only developers

**Status:** Accepted

---

### ADR-010: Haptic Feedback via Expo SDK (No Haptics Polyfill)

**Decision:** Use `expo-haptics` (built into Expo SDK) for device vibration in the Draw module winner announcement.

**Context:**
- Expo SDK version 49+ includes Haptics API out of the box
- No need for external `react-native-vibration` or polyfills
- Standardized haptic patterns (light, medium, heavy) map to draw feature needs

**Alternatives Considered:**
1. **react-native-vibration** – Older, less maintained; Expo Haptics is superset
2. **Native haptics API directly** – Requires native module bridge; Expo abstracts this

**Trade-offs Accepted:**
- Tied to Expo SDK version (upgrade required for haptics improvements)

**Status:** Accepted

## 3. System Components

### 3.1 Dice Module

**Responsibility:** Generate random dice rolls (1–6 values), display animated dice, show total sum.

**Key Interfaces:**
- **Consumed:**
  - Navigation context (access to current module state)
  - Theme context (colors, spacing)
- **Exposed:**
  - `DiceScreen` component (rendered in tab navigator)
  - `useDiceRoll()` hook (roll state and result)

**Dependencies:**
- `react-native-reanimated` (dice spinning animation)
- Custom `random` utility (uniform 1–6 generation)

**Internal Structure:**
- `DiceScreen.tsx` – Main entry point
- `DiceDisplay.tsx` – Visual display of individual dice
- `DiceRoller.tsx` – Controls (quantity selector, roll button)
- `useDiceRoll.ts` – State hook (quantity, result, rolling flag)

---

### 3.2 Score Module

**Responsibility:** Manage game sessions, player roster, score tracking, and persistence.

**Key Interfaces:**
- **Consumed:**
  - `GameRepository` (load/save games)
  - Navigation context (open/close game editor)
  - Theme context
- **Exposed:**
  - `ScoreScreen` component
  - `useGame(gameId)` hook (load specific game)
  - `useGameList()` hook (list all games)

**Dependencies:**
- `StorageService` / `GameRepository`
- Custom `validation` utility (name length, player count)

**Internal Structure:**
- `ScoreScreen.tsx` – Main entry point (shows game list)
- `GameEditor.tsx` – Create/edit game
- `GameList.tsx` – List of saved games
- `PlayerRow.tsx` – Individual player score row
- `useGame.ts` – Load and update a single game
- `useGamePersistence.ts` – Wrapper around GameRepository

---

### 3.3 Draw Module

**Responsibility:** Multi-touch raffle input, winner selection, result animation.

**Key Interfaces:**
- **Consumed:**
  - Navigation context
  - `useMultiTouch()` hook (touch detection)
  - Theme context
- **Exposed:**
  - `DrawScreen` component
  - `useMultiTouch()` hook (touch points, winner)

**Dependencies:**
- `react-native-gesture-handler` (multi-touch detection)
- `react-native-reanimated` (circle animations)
- `expo-haptics` (vibration on winner)
- Custom `random` utility (uniform winner selection)

**Internal Structure:**
- `DrawScreen.tsx` – Main entry point
- `TouchArea.tsx` – Gesture detector + circle renderer
- `DrawResult.tsx` – Winner announcement and animation
- `useMultiTouch.ts` – Touch point state management

---

### 3.4 Persistence Service Layer

**Responsibility:** Serialize/deserialize game data to/from AsyncStorage.

**Key Interfaces:**
- **Exposed:**
  - `StorageService` – low-level key-value API
  - `GameRepository` – high-level game CRUD API

**Dependencies:**
- `@react-native-async-storage/async-storage`

**Internal Structure:**
- `StorageService.ts` – Wrapper around AsyncStorage (encryption, error handling in future)
- `GameRepository.ts` – Game-specific queries (list all, load by ID, save, delete)

---

### 3.5 Navigation System

**Responsibility:** Tab navigation between three modules, pass state between screens.

**Key Interfaces:**
- **Consumed:**
  - Module screens (Dice, Score, Draw)

- **Exposed:**
  - `RootNavigator` component (root of navigation tree)
  - `NavigationContext` (current tab, active game ID)

**Internal Structure:**
- `RootNavigator.tsx` – BottomTabNavigator with three stacks
- `navigationType.ts` – TypeScript types for navigation params

**Responsibility:** Centralize colors, spacing, typography, and accessibility settings.

**Key Interfaces:**

  - `colors` object (primary, secondary, backgrounds)

**Internal Structure:**

## 6. High-Level Data Model

### Entities

| Entity | Storage | Fields | Notes |
|--------|---------|--------|-------|
| **GameSession** | AsyncStorage (key: `game_${id}`) | `id` (UUID), `createdAt` (timestamp), `lastAccessedAt` (timestamp), `players` (Player[]), `status` ('active' \| 'archived') | One entry per game session. Max 8 players. No schema versioning in v1. |
| **Player** | Nested in GameSession | `id` (UUID), `name` (string, 1–30 chars), `score` (number, ≥ 0) | No separate storage; always within GameSession. |
| **GameIndex** | AsyncStorage (key: `game_index`) | Array of `{id, createdAt, lastAccessedAt}` | Denormalized for fast listing. Synced on game save/delete. |
| **DiceRoll** | **Not persisted** | `dice` (number[]), `sum` (number), `timestamp` (optional) | Ephemeral; discarded after animation. |
| **DrawResult** | **Not persisted** | `touchPoints` (TouchPoint[]), `winner` (TouchPoint), `timestamp` (optional) | Ephemeral; discarded after animation. |
| **TouchPoint** | Ephemeral (Draw module only) | `id` (string), `x` (number), `y` (number), `color` (string) | Not persisted. Used during multi-touch input. |

### Key Relationships

- GameSession **has many** Players (1:N)
- GameIndex **references** GameSession by ID (denormalization for listing)
- DiceRoll, DrawResult, TouchPoint are **transient**; no persistent relationships

### Storage Locations

```typescript
// Game data in AsyncStorage
AsyncStorage.setItem('game_${gameId}', JSON.stringify(gameSession));
AsyncStorage.setItem('game_index', JSON.stringify(indexArray));

// No other persisted data
```

All interfaces are TypeScript types; no HTTP requests.

```typescript
// GameRepository (persistence/GameRepository.ts)
interface GameRepository {
  listGames(): Promise<GameSession[]>;
  getGame(id: string): Promise<GameSession | null>;
  saveGame(game: GameSession): Promise<void>;
  deleteGame(id: string): Promise<void>;
  createGame(players: string[]): Promise<GameSession>;
}

// useGame hook (hooks/useGame.ts)
function useGame(gameId: string): {
  game: GameSession | null;
  loading: boolean;
  error: Error | null;
  updateScore(playerId: string, delta: number): Promise<void>;
  updateName(playerId: string, newName: string): Promise<void>;
};

// useGameList hook (hooks/useGameList.ts)
function useGameList(): {
  games: GameSession[];
  loading: boolean;
  error: Error | null;
  refresh(): Promise<void>;
};
```

## 8. Open Questions

### 1. **Negative Score Handling (Critical for UX)**
   - Requirement FA-03.2 presents two options: allow negative scores or block at zero.
   - **Decision Needed:** Which behavior is intended? (Current spec: ambiguous.)
   - **Implication:** Changes validation logic in `PlayerRow` and `useGame` hook.

### 2. **Game Closure / Archive State**
   - Is there a concept of "end game" or "archive game"?
   - Can users continue editing a game after marking it finished?
   - **Implication:** Affects game status enum and UI (show "active" vs. "archived" separately?).

### 3. **Historical Score Change Log**
   - Spec excludes "historial de partidas," but does this mean:
     - No summary stats of past partidas? (Accepted: exclusion)
     - Or no log of individual score deltas within a partida? (Ambiguous)
   - **Implication:** If deltas must be logged (for undo feature, future audit), data model changes.

### 4. **Puntaje Overflow and Bounds**
   - What is the maximum displayable score?
   - JavaScript Number can hold `2^53 - 1` (~9 quadrillion), but UI may truncate.
   - **Implication:** Add input validation and display truncation if needed.

### 5. **Multi-Device Sync (Future-Proofing)**
   - Spec excludes cross-device sync for v1, but should the data model be designed to allow it in v2?
   - E.g., should GameSession have a UUID instead of random ID?
   - **Implication:** Minimal overhead to use UUID now; plan for conflict resolution later.

### 6. **Touch Pressure Sensitivity (3D Touch / Force Touch)**
   - Does the Draw module care about touch pressure, or only position?
   - iOS 3D Touch and Android pressure sensors provide extra data.
   - **Implication:** Probably ignore pressure for v1 (spec is silent); design to add later if needed.

### 7. **Draw Module: New Touch During Stability Count**
   - Spec FA-05.2 presents two options:
     - Option A: New touch resets counter (more forgiving, more chaos)
     - Option B: New touch is ignored (strict, easier to test)
   - **Decision Needed:** Which option is the intended UX?

### 8. **Data Migration and Schema Versioning**
   - If GameSession schema changes in v1.1, how do we migrate old game data?
   - Should we add a `schemaVersion` field to GameSession now?
   - **Implication:** Add versioning helper in `GameRepository` before v1.0 release.

### 9. **Maximum Number of Saved Games**
   - Spec says "without limit specified."
   - AsyncStorage has ~10 MB soft limit. Average game session is ~200–500 bytes (2–8 players).
   - Practically: can store 20k–50k games before hitting limit.
   - **Question:** Is this acceptable, or should we add a practical hard limit (e.g., 1000 games) with auto-cleanup?

### 10. **Localization / Internationalization**
   - Spec specifies "Spanish only for MVP."
   - Should the code be structured for future i18n (i.e., use string constants, not hardcoded Spanish text)?
   - **Implication:** Minimal overhead to structure now; worth doing for future translations.

### 11. **Dark Mode / Theme Switching**
   - Spec says "theme visual único en v1" (single visual theme).
   - Should the theme object be designed to support multiple themes (light/dark) for future?
   - **Implication:** Yes, design theme as swappable even if v1 only implements one.

### 12. **Accessibility: Screen Reader Support**
   - Spec requires WCAG AA contrast and dynamic type, but does it require screen reader labels?
   - iOS VoiceOver and Android TalkBack are standard; should we add `accessibilityLabel` to all buttons?
   - **Implication:** Recommend adding for v1; minimal effort, large accessibility impact.

## 9. Implementation Notes

### 9.1 Expo Managed Workflow
- Use `expo init` to scaffold project (not `npx create-expo-app`; gives better defaults).
- Managed workflow means no native module editing; everything via Expo SDK or community modules.
- For custom native code beyond Expo, move to bare workflow (later if needed).

### 9.2 Build and Deployment
- Use **Expo EAS Build** for managed iOS and Android builds.
- Configure in `eas.json`:
  - Separate build profiles for development and release.
  - Release builds include code obfuscation and optimization.
- Deploy to App Store and Google Play manually (or via EAS Submit in future).

### 9.3 Testing Strategy
- **Unit tests:** Hooks (`useDiceRoll`, `useGame`), utils (`random`, `validation`), services (`GameRepository`).
- **Component tests:** Individual UI components with React Native Testing Library.
- **Integration tests:** Full user flows (create game → add score → save).
- **E2E (optional for v1):** Detox for full app flow testing on real simulators.

### 9.4 Performance Tuning
- Monitor bundle size: `react-native-bundle-analyzer` or EAS Build logs.
- Lazy-load module screens if bundle approaches 5 MB (acceptable limit for app stores).
- Use `React.memo` for expensive components (not necessary for v1 unless measured).
- Profile animations: use React Native DevTools `Performance` tab.

### 9.5 Error Handling
- All async operations (persist, load) wrapped in try-catch.
- User-facing errors shown as toast or modal (not console spam).
- Log errors to console in dev; consider Sentry or similar for production (v2).

### 9.6 Accessibility Compliance
- Test with iOS VoiceOver and Android TalkBack enabled.
- Use `testID` props for testing; `accessibilityLabel` for screen readers.
- Color palette: ensure 4.5:1 contrast ratio (WCAG AA) for all text.
- Minimum touch target: 48x48 dp (Material Design standard).
