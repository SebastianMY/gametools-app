# ESPECIFICACIÓN FUNCIONAL
## App Móvil de Utilidades para Juegos de Mesa y Cartas

---

## 1. RESUMEN DEL PROYECTO

**Nombre del Proyecto:** Game Table Companion

**Versión:** 1.0

**Fecha de Documento:** 2026-03-18

**Descripción General:**
Aplicación móvil multiplataforma (iOS y Android) que proporciona herramientas esenciales para jugadores de juegos de mesa y cartas. La app reemplaza objetos físicos comúnmente utilizados (dados, marcadores de puntaje, métodos de sorteo) mediante una interfaz visual intuitiva inspirada en la experiencia de una mesa de juego real. Diseñada para funcionar completamente offline, sin requerir registro, login o conexión a servidor backend.

**Público Objetivo:**
Jugadores casuales de juegos de mesa y cartas de todas las edades (6+), incluyendo familias, amigos y grupos sociales.

---

## 2. OBJETIVOS Y ALCANCE

### 2.1 Objetivos Principales

1. **Reemplazar elementos físicos:** Proporcionar herramientas digital para dados, anotación de puntajes y sorteos.
2. **Experiencia sin fricción:** Permitir uso inmediato sin autenticación, validación de email o requisitos previos.
3. **Persistencia de datos:** Guardar automáticamente partidas en el dispositivo para retomar sesiones posteriores.
4. **Accesibilidad intuitiva:** Diseñar una interfaz visual clara e inspirada en contextos reales de juego.
5. **Rendimiento consistente:** Garantizar animaciones fluidas y respuesta inmediata a interacciones táctiles.

### 2.2 Alcance Funcional (v1)

**Incluido:**
- Tirador de dados (1-6 dados simultáneos)
- Gestor de puntajes multipartida con hasta 8 jugadores
- Sistema de sorteo multitáctil con visualización en tiempo real
- Almacenamiento local persistente de partidas
- Interfaz responsive para teléfonos y tablets
- Accesibilidad visual (contraste WCAG AA, tipografía dinámica)

**Excluido (v1):**
- Autenticación de usuarios
- Sincronización en la nube entre dispositivos
- Reglas específicas de juegos (Truco, UNO, Póker, etc.)
- Modo multijugador en red
- Monetización (publicidad, compras in-app)
- Estadísticas de histórico de partidas
- Modos de juego predefinidos

## 4. CASOS DE USO PRINCIPALES

### 4.1 Caso de Uso: Tirar Dados (CU-01)

**Identificador:** CU-01-DiceThrowing

**Título:** Tirar dados y visualizar resultado

**Actores Primarios:** Usuario Anónimo

**Precondiciones:**
- La app está abierta
- El usuario se encuentra en la sección "Dados"

**Flujo Principal:**

1. El usuario visualiza la pantalla de Dados con un selector de cantidad
2. El usuario toca el control para seleccionar la cantidad de dados (1 a 6)
3. El sistema valida que la selección esté dentro del rango permitido
4. El usuario toca el botón "Tirar" o equivalente
5. El sistema inicia animación de giro en todos los dados (efecto visual durante 0.8-1.2 segundos)
6. Al finalizar la animación, el sistema genera números aleatorios (1-6) para cada dado
7. Se despliegan visualmente los resultados:
   - Cada dado muestra su número individual
   - Se calcula y muestra el total de la suma
8. El usuario puede tirar nuevamente repitiendo desde el paso 2

**Flujos Alternativos:**

**FA-01.1: El usuario no selecciona cantidad de dados**
- Precondición: El usuario abre la sección de Dados
- Acción: Si no hay selección, el sistema asume 1 dado por defecto
- Resultado: Se muestra opción de cambiar a través del selector

**FA-01.2: El usuario desea cambiar cantidad antes de tirar**
- Acción: El usuario toca el selector de cantidad nuevamente
- Resultado: Se actualiza la selección y está lista para el próximo tiro

**Postcondiciones:**
- Los resultados se muestran en pantalla
- El usuario puede tirar nuevamente sin límite de intentos
- La acción no requiere persistencia (no se guardan resultados de dados individuales)

**Requisitos Especiales:**
- Animación debe ejecutarse a 60 fps
- Duración de animación: 0.8-1.2 segundos para efecto de spinning
- Cada dado debe mostrar claramente su número individual
- La suma total debe ser visible y legible

### 4.2 Caso de Uso: Crear Partida de Puntaje (CU-02)

**Identificador:** CU-02-CreateScoreGame

**Título:** Crear nueva partida con múltiples jugadores

**Actores Primarios:** Usuario Anónimo

**Precondiciones:**
- La app está abierta
- El usuario se encuentra en la sección "Puntaje"
- No existe una partida activa abierta

**Flujo Principal:**

1. El usuario visualiza la pantalla de Puntaje con opción "Nueva Partida"
2. El usuario toca el botón "Nueva Partida"
3. El sistema presenta un formulario para ingresar nombre de jugadores
4. El usuario ingresa el nombre del primer jugador (campo de texto editable)
5. El usuario agrega más jugadores tocando "+ Agregar Jugador" (hasta máximo 8)
6. Para cada jugador adicional, repite los pasos 4-5
7. El usuario revisa la lista de jugadores (2-8 nombres ingresados)
8. El usuario toca "Iniciar Partida" para confirmar
9. El sistema:
   - Valida que al menos 2 jugadores estén ingresados
   - Inicializa puntaje en 0 para cada jugador
   - Crea entrada en almacenamiento local con timestamp de creación
   - Navega a la pantalla de gestión de puntaje
10. La pantalla muestra todos los jugadores con su nombre y puntaje actual (0)

**Flujos Alternativos:**

**FA-02.1: Usuario intenta crear partida con solo 1 jugador**
- Acción: El usuario toca "Iniciar Partida" con un único jugador
- Resultado: Sistema muestra mensaje de error "Se requieren al menos 2 jugadores"
- Acción: Usuario debe agregar al menos uno más

**FA-02.2: Usuario intenta agregar más de 8 jugadores**
- Acción: El usuario toca "+ Agregar Jugador" cuando ya hay 8
- Resultado: El botón está deshabilitado o muestra mensaje informativo
- Posibilidad: El usuario puede reemplazar un nombre existente

**FA-02.3: Usuario deja campo de nombre vacío**
- Acción: El usuario intenta avanzar sin completar un nombre
- Resultado: El campo se resalta con indicación de error "Campo obligatorio"

**FA-02.4: El usuario abandona sin iniciar**
- Acción: El usuario navega hacia atrás sin tocando "Iniciar Partida"
- Resultado: Se descarta el formulario sin guardar

**Postcondiciones:**
- Nueva partida creada en almacenamiento local
- Partida aparece en lista de "Partidas Guardadas"
- Usuario navega automáticamente a pantalla de gestión de puntaje
- Partida está en estado activo

**Requisitos Especiales:**
- Máximo 8 jugadores por partida
- Mínimo 2 jugadores para iniciar
- Nombres de jugadores con máximo 30 caracteres
- Almacenamiento debe incluir timestamp de creación y último acceso

### 4.3 Caso de Uso: Modificar Puntaje de Jugadores (CU-03)

**Identificador:** CU-03-UpdateScore

**Título:** Sumar o restar puntos a jugadores durante una partida activa

**Actores Primarios:** Usuario Anónimo

**Precondiciones:**
- La app está abierta
- Existe una partida activa en la sección "Puntaje"
- Se visualiza la pantalla de gestión de puntaje con lista de jugadores

**Flujo Principal:**

1. El usuario visualiza la pantalla con lista de jugadores y sus puntajes actuales
2. El usuario identifica al jugador cuyo puntaje desea modificar
3. El usuario toca el botón "+" (sumar) o "−" (restar) junto al puntaje del jugador
4. El sistema registra la acción:
   - Incrementa o decrementa puntaje en 1 punto
   - Valida que el puntaje no sea menor a 0 (opcional: permitir negativos)
   - Actualiza la visualización del puntaje
   - Guarda el cambio automáticamente en almacenamiento local
5. El puntaje actualizado se muestra inmediatamente en pantalla
6. El usuario puede continuar modificando puntajes de otros jugadores o del mismo
7. El usuario puede retomar esta partida más tarde y verá los puntajes exactos

**Flujos Alternativos:**

**FA-03.1: Usuario desea incrementar puntaje en más de 1 punto**
- Precondición: Usuario presiona "+" múltiples veces rápidamente
- Acción: Cada toque incrementa en 1
- Resultado: Puntaje se incrementa por cada toque (sin límite de velocidad)

**FA-03.2: El usuario intenta restar a un puntaje de 0**
- Acción: El usuario toca "−" cuando puntaje = 0
- Resultado Opción A: El puntaje pasa a −1 (permitir negativos)
- Resultado Opción B: El sistema no realiza cambio y muestra indicador visual (recomendado para UX)

**FA-03.3: Usuario presiona larga duración (long press) en botón +/−**
- Acción: El usuario mantiene presionado el botón + o −
- Resultado: El puntaje se incrementa/decrementa continuamente mientras se mantiene presión (incremento acelerado)

**FA-03.4: El usuario desea editar manualmente el puntaje**
- Acción: El usuario toca directamente en el número del puntaje
- Resultado: Se abre teclado numérico o campo de entrada para ingresar valor exacto
- Validación: Solo números, máximo 5 dígitos, mínimo 0 (o según FA-03.2)

**Postcondiciones:**
- Puntaje se actualiza en tiempo real
- Cambio se persiste automáticamente en almacenamiento local
- No se requiere confirmación adicional
- La partida mantiene su estado exacto

**Requisitos Especiales:**
- Área táctil mínima de 48x48dp para botones +/−
- Actualización debe ser instantánea (< 100ms)
- Persistencia después de cada cambio individual

### 4.4 Caso de Uso: Restaurar Partida Guardada (CU-04)

**Identificador:** CU-04-ResumeGame

**Título:** Retomar una partida previamente guardada

**Actores Primarios:** Usuario Anónimo

**Precondiciones:**
- La app está abierta
- Existen partidas guardadas en almacenamiento local
- El usuario se encuentra en la sección "Puntaje"

**Flujo Principal:**

1. El usuario visualiza la pantalla de Puntaje con sección "Partidas Guardadas"
2. El sistema lista todas las partidas guardadas ordenadas por:
   - Partidas activas primero (sin fecha de cierre)
   - Ordenadas por fecha de último acceso (más recientes primero)
3. Cada entrada de partida muestra:
   - Nombre/Identificador de la partida (ej: "Partida del 15/3")
   - Nombres de jugadores resumidos
   - Fecha de creación y último acceso
4. El usuario toca sobre una partida de la lista
5. El sistema carga los datos exactos:
   - Restaura todos los nombres de jugadores
   - Restaura puntajes exactos de cada jugador
   - Restaura cualquier otro estado asociado
6. La pantalla de gestión de puntaje se muestra con la partida completamente restaurada
7. El usuario puede continuar modificando puntajes desde donde la dejó

**Flujos Alternativos:**

**FA-04.1: No hay partidas guardadas**
- Precondición: El usuario abre la sección Puntaje sin partidas previas
- Resultado: Se muestra mensaje "No hay partidas guardadas" con botón "Nueva Partida"

**FA-04.2: Usuario desea eliminar una partida guardada**
- Acción: El usuario desliza (swipe) o mantiene presionada una partida de la lista
- Resultado: Se abre menú contextual con opción "Eliminar"
- Confirmación: Se solicita confirmación antes de eliminar permanentemente

**FA-04.3: El usuario crea una nueva partida mientras una está activa**
- Acción: El usuario toca "Nueva Partida"
- Resultado: La partida anterior se guarda automáticamente
- Comportamiento: Ambas partidas coexisten en la lista

**Postcondiciones:**
- Partida está en pantalla lista para continuar editando
- Timestamp de "último acceso" se actualiza
- Todos los datos se cargan correctamente

**Requisitos Especiales:**
- Restauración debe ser inmediata (< 200ms)
- Mantener exactitud de datos (sin redondeos)
- Soporte para coexistencia de múltiples partidas (sin límite especificado)

### 4.5 Caso de Uso: Realizar Sorteo Multitáctil (CU-05)

**Identificador:** CU-05-MultiTouchDrawing

**Título:** Realizar sorteo entre múltiples jugadores mediante input multitáctil

**Actores Primarios:** Usuario Anónimo

**Precondiciones:**
- La app está abierta
- El usuario se encuentra en la sección "Sorteo"
- Dispositivo soporta multitáctil (requiere capacidad de al menos 5-10 puntos de contacto simultáneos)

**Flujo Principal:**

1. El usuario visualiza la pantalla de Sorteo con instrucciones visuales:
   - "Cada jugador apoya un dedo en la pantalla"
   - Área visual clara (ej: fondo con patrón de mesa)
2. El usuario (o grupo de jugadores) toca la pantalla con múltiples dedos simultáneamente
3. Para cada punto de contacto detectado, el sistema:
   - Asigna un color único y diferenciado del círculo visual
   - Dibuja un círculo en la posición del toque
   - Mantiene el círculo mientras el dedo está en contacto
4. Mientras se mantienen los toques:
   - Los círculos permanecen visibles con sus colores
   - El sistema contabiliza 3 segundos de estabilidad
   - Contador visual muestra los segundos restantes
5. Tras 3 segundos de estabilidad:
   - El dispositivo vibra con patrón de retroalimentación (150ms + pausa + 100ms)
   - El sistema selecciona aleatoriamente uno de los dedos/colores como ganador
6. Animación de resultado:
   - El círculo ganador crece con efecto de escala (0.5s de duración)
   - Los círculos no ganadores desaparecen con fade-out (0.3s)
   - Se muestra brevemente un indicador visual del ganador
7. El usuario puede reiniciar el sorteo tocando "Sorteo nuevamente" o similar

**Flujos Alternativos:**

**FA-05.1: Se retira un dedo antes de los 3 segundos**
- Acción: Un jugador retira el dedo durante el conteo
- Resultado: El círculo correspondiente desaparece inmediatamente
- Comportamiento: El contador se reinicia desde 0 segundos
- Nota: Solo se consideran los dedos que permanecen contacto después de reinicio

**FA-05.2: Se agrega un nuevo dedo después de iniciado el conteo**
- Acción: Un nuevo jugador añade su dedo después de iniciado el conteo de 3s
- Resultado Opción A: El nuevo dedo se agrega y el contador se reinicia
- Resultado Opción B: El nuevo dedo se ignora (solo cuenta dedos iniciales)
- Recomendación: Opción A para mayor flexibilidad

**FA-05.3: Solo 1 dedo toca la pantalla**
- Acción: Un único jugador toca la pantalla (sorteo entre 1 persona)
- Resultado: Sistema muestra mensaje "Se requieren al menos 2 dedos/jugadores"
- Comportamiento: No inicia el contador de 3 segundos

**FA-05.4: El usuario desea cancelar el sorteo**
- Acción: El usuario retira todos los dedos antes de finalizar
- Resultado: Se limpia la pantalla y se resetea el estado

**FA-05.5: Múltiples dedos de la misma persona (no se desea)**
- Escenario: Una persona apoya varios dedos
- Comportamiento: Cada dedo se trata como entrada independiente
- Nota: No hay validación para prevenir esto (es responsabilidad del grupo)

**Postcondiciones:**
- Un ganador ha sido seleccionado visualmente
- Animación de ganador finaliza completamente
- Usuario puede iniciar nuevo sorteo
- No se persisten resultados del sorteo (estateless)

**Requisitos Especiales:**
- Soporte multitáctil con capacidad de mínimo 10 toques simultáneos
- Detección de contacto con latencia < 50ms
- Colores de círculos deben ser distinguibles entre sí (8+ colores únicos disponibles)
- Vibración del dispositivo: patrón táctico (150ms + 50ms + 100ms)
- Contador visual de 3 segundos debe ser muy legible
- Selección aleatoria debe ser uniforme (no sesgada)

## 10. CRITERIOS DE ACEPTACIÓN

### 10.1 Criterios de Aceptación: Módulo de Dados

- [ ] Usuario selecciona 1-6 dados y ve selector funcional
- [ ] Al tocar "Tirar", animación ejecuta suavemente durante 0.8-1.2 segundos
- [ ] Cada dado muestra un número final entre 1-6
- [ ] Suma total es correcta y visible
- [ ] Presionar "Tirar" nuevamente sin delay; no hay persistencia

### 10.2 Criterios de Aceptación: Módulo de Puntaje

- [ ] Crear partida: ingresar 2-8 nombres, todos inician en 0 puntos
- [ ] Botones +/− funcionan correctamente; puntaje actualiza en tiempo real
- [ ] Cambios persisten al cerrar y reabrir la app
- [ ] Lista de partidas guardadas es accesible y ordenada por fecha
- [ ] Al tocar una partida, se restaura exactamente su último estado
- [ ] Múltiples partidas pueden coexistir sin conflictos

### 10.3 Criterios de Aceptación: Módulo de Sorteo

- [ ] Multitáctil detecta 2-10 dedos simultáneos
- [ ] Cada dedo es un círculo de color único
- [ ] Contador de 3 segundos es visible
- [ ] Al retirar/agregar dedo, contador se reinicia
- [ ] Tras 3s, vibración y selección de ganador
- [ ] Círculo ganador crece; otros desaparecen

### 10.4 Criterios de Aceptación: Interfaz General

- [ ] App abre en < 2 segundos
- [ ] Navegación entre Dados, Puntaje, Sorteo es fluida
- [ ] Todos los botones son tocables (48x48dp mínimo)
- [ ] Contraste cumple WCAG AA
- [ ] Diseño visual es inspirado en mesa de juego real

### 10.5 Criterios de Aceptación: Performance

- [ ] Animaciones a 60 fps constantes
- [ ] Respuesta táctil < 100 ms
- [ ] Persistencia < 50 ms
- [ ] Carga de partida < 200 ms
- [ ] App funciona 100% sin internet

## 11. GLOSARIO Y DEFINICIONES

| Término | Definición |
|---------|-----------|
| **Partida** | Sesión de juego con múltiples jugadores, cada uno con un puntaje. Se guarda en el dispositivo. |
| **Jugador** | Participante en una partida, identificado por nombre, con puntaje asociado. |
| **Tirada** | Acción de "tirar" dados una sola vez, generando números aleatorios para cada dado. |
| **Sorteo** | Acción de seleccionar un ganador al azar entre múltiples dedos tocando simultáneamente la pantalla. |
| **Persistencia** | Guardado automático de datos en almacenamiento local del dispositivo. |
| **Offline** | Funcionamiento sin conexión a internet. |
| **Asyncstorage** | Sistema de almacenamiento key-value nativo en React Native. |
| **SQLite** | Base de datos relacional embebida en dispositivos móviles. |
| **Expo** | Framework que simplifica desarrollo de apps React Native. |
| **EAS Build** | Servicio en nube de Expo para compilar apps para iOS y Android. |
| **Multitáctil / Multi-touch** | Capacidad de la pantalla de detectar múltiples toques simultáneos. |
| **Haptics / Haptic Feedback** | Retroalimentación táctil (vibración) del dispositivo. |
| **Estabilidad (Sorteo)** | Período de 3 segundos sin cambios en los dedos tocando la pantalla. |
