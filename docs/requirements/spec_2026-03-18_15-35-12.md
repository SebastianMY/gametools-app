# Especificación Funcional: Game Tools Mobile App

## 1. Resumen del Proyecto

**Game Tools** es una aplicación móvil nativa (iOS y Android) desarrollada con React Native y Expo que reemplaza herramientas físicas comúnmente utilizadas en juegos de mesa y cartas: dados, anotador de puntaje y métodos de sorteo. 

La aplicación está dirigida a jugadores de todas las edades y proporciona una experiencia visual intuitiva inspirada en una mesa de juego real, con una estética temática verde, dados 3D, fichas coloridas y tipografía manuscrita. Funciona completamente offline sin requerir autenticación, registro o conexión a servidor, permitiendo a cualquier usuario acceder inmediatamente a todas las funcionalidades.

---

## 2. Objetivos y Alcance

### 2.1 Objetivos Principales

- Proporcionar una solución digital integrada para las herramientas más comunes en juegos de mesa y cartas.
- Ofrecer una interfaz intuitiva y visualmente atractiva que emule la experiencia física de una mesa de juego.
- Garantizar persistencia de datos local sin dependencias de infraestructura en la nube.
- Soportar multijugadores locales mediante interacción táctil simultanea.
- Permitir la continuidad de partidas entre sesiones del usuario.

### 2.2 Alcance (v1)

**Incluido:**
- Generador de dados (1-6 dados simultáneamente).
- Anotador de puntaje para hasta 8 jugadores.
- Herramienta de sorteo multitáctil.
- Persistencia local de partidas.
- Interfaz para iOS y Android con código unificado.
- Accesibilidad WCAG AA.

**No incluido:**
- Sistemas de autenticación o cuentas de usuario.
- Sincronización entre dispositivos.
- Reglas de juegos específicos.
- Modo multijugador en red o multiplayer online.
- Monetización, compras in-app o publicidad.
- Leaderboards globales o estadísticas en la nube.

## 4. Casos de Uso Principales

### 4.1 Caso de Uso: Tirar Dados

**Identificador:** UC-001  
**Nombre:** Tirar Dados  
**Actor Primario:** Usuario Anónimo  
**Precondiciones:** La aplicación está abierta en la sección Dados.  

#### Flujo Principal

1. El usuario visualiza la pantalla de Dados con un selector de cantidad.
2. El usuario selecciona la cantidad de dados a tirar (entre 1 y 6) mediante un control visual (ej: botones +/- o slider).
3. El usuario toca el botón "Tirar" o realiza una acción táctil desencadenante.
4. Los dados inician una animación de giro/rotación que simula el lanzamiento físico (duración: 1-2 segundos).
5. Tras completar la animación, se muestran los resultados:
   - Cada dado muestra su valor individual (1-6).
   - Se exhibe la suma total de los dados.
6. Los resultados permanecen en pantalla hasta que el usuario realiza un nuevo lanzamiento.

#### Flujo Alternativo A: Modificar cantidad de dados

1. Si el usuario modifica la cantidad antes de tocar "Tirar", el selector se actualiza.
2. El nuevo lanzamiento utiliza la cantidad modificada.

#### Flujo Alternativo B: Visualizar historial (futuro)

1. Si se implementa, se guardaría el último lanzamiento para referencia rápida.

**Postcondiciones:** Los resultados del lanzamiento se muestran en pantalla; no se guardan en persistencia (dado que no es data de partida).

### 4.2 Caso de Uso: Crear Partida de Puntaje

**Identificador:** UC-002  
**Nombre:** Crear Partida de Puntaje  
**Actor Primario:** Usuario Anónimo  
**Precondiciones:** La aplicación está en la sección Puntaje. El usuario no tiene una partida activa o desea crear una nueva.

#### Flujo Principal

1. El usuario toca el botón "Nueva Partida" o "Crear".
2. Se abre un formulario de creación donde ingresa:
   - Nombres de los jugadores (campo de texto individual por jugador).
   - Cantidad de jugadores (entre 2 y 8).
   - (Opcional) Puntaje inicial para todos (ej: 0, 100, 500, etc.).
   - (Opcional) Nombre de la partida para identificarla.
3. El usuario confirma los datos tocando "Iniciar" o "Comenzar".
4. La aplicación crea la partida con estado inicial (cada jugador con su puntaje inicial en 0 o el valor especificado).
5. Se guarda automáticamente en almacenamiento local (AsyncStorage o SQLite).
6. Se muestra la pantalla de la partida con una tabla de jugadores y sus puntajes.

#### Flujo Alternativo A: Cancelar creación

1. El usuario toca "Cancelar" antes de confirmar.
2. Vuelve a la pantalla anterior sin crear la partida.

#### Flujo Alternativo B: Cargar partida anterior

1. En lugar de crear nueva, el usuario ve una lista de partidas guardadas.
2. Selecciona una partida anterior.
3. Se restaura el estado exacto de esa partida.

**Postcondiciones:** Una nueva partida está creada y lista para recibir cambios de puntaje. Se persiste en el almacenamiento local.

### 4.3 Caso de Uso: Modificar Puntaje de Jugador

**Identificador:** UC-003  
**Nombre:** Modificar Puntaje de Jugador  
**Actor Primario:** Usuario Anónimo  
**Precondiciones:** Una partida está activa. Existe al menos un jugador en la partida.

#### Flujo Principal

1. El usuario visualiza la pantalla de Puntaje con la lista de jugadores y sus puntajes actuales.
2. Junto a cada jugador hay botones de suma (+) y resta (−).
3. El usuario toca el botón "+" para incrementar el puntaje, o "−" para decrementarlo.
   - Cada toque suma o resta una cantidad predeterminada (ej: 1 punto, 5 puntos, 10 puntos, configurable).
4. El puntaje del jugador se actualiza inmediatamente en la UI.
5. La partida se guarda automáticamente sin retrasos perceptibles.
6. El usuario puede continuar modificando el puntaje de otros jugadores o del mismo jugador.

#### Flujo Alternativo A: Entrada de puntaje manual

1. El usuario toca directamente en el campo de puntaje del jugador.
2. Se abre un teclado numérico o campo de entrada de texto.
3. El usuario ingresa el nuevo valor de puntaje.
4. Al confirmar, el puntaje se actualiza y se persiste.

#### Flujo Alternativo B: Incremento personalizado

1. Si existe un selector de incremento, el usuario lo modifica (ej: cambiar de +1 a +5).
2. Los siguientes toques en "+" o "−" utilizan el nuevo incremento.

**Postcondiciones:** El puntaje del jugador es actualizado y persistido localmente de inmediato.

---

### 4.4 Caso de Uso: Retomar Partida Guardada

**Identificador:** UC-004  
**Nombre:** Retomar Partida Guardada  
**Actor Primario:** Usuario Anónimo  
**Precondiciones:** El usuario cerró la app previamente. Existe al menos una partida guardada en el almacenamiento local.

#### Flujo Principal

1. El usuario reabre la aplicación.
2. Al entrar a la sección Puntaje, ve una lista de partidas guardadas con:
   - Nombre de la partida (o nombre de los jugadores).
   - Puntajes actuales de cada jugador.
   - Fecha/hora de la última actualización.
   - Opción de continuar o eliminar.
3. El usuario selecciona la partida que desea retomar tocando en ella.
4. La aplicación restaura el estado exacto de la partida (nombres de jugadores, puntajes, cualquier otra configuración).
5. Se muestra la pantalla de juego y el usuario puede continuar modificando puntajes.

#### Flujo Alternativo A: Crear nueva en lugar de retomar

1. El usuario toca "Nueva Partida" en lugar de seleccionar una guardada.
2. Sigue el flujo UC-002.

#### Flujo Alternativo B: Eliminar partida antigua

1. El usuario puede deslizar o tocar una opción de eliminar en la partida guardada.
2. Se pide confirmación antes de eliminar.
3. Se elimina de la persistencia local.

**Postcondiciones:** La partida está activa con su estado previo completamente restaurado. El usuario puede continuar jugando.

### 4.5 Caso de Uso: Realizar Sorteo Multitáctil

**Identificador:** UC-005  
**Nombre:** Realizar Sorteo Multitáctil  
**Actor Primario:** Usuario Anónimo (grupo de jugadores)  
**Precondiciones:** La aplicación está en la sección Sorteo. Hay al menos 2 dispositivos con dedo en pantalla o el dispositivo soporta multitáctil simultáneo.

#### Flujo Principal

1. El usuario abre la sección Sorteo.
2. Se muestra una pantalla vacía/lista lista para recibir toques.
3. Se visualiza un instructivo: "Todos apoyen un dedo en la pantalla simultáneamente".
4. Cada jugador apoya su dedo en un área distinta de la pantalla.
5. Cada toque se representa con un círculo de color único (sistema de asignación de colores automático).
6. El círculo persiste mientras el dedo permanece en contacto.
7. Tras 3 segundos de estabilidad (sin nuevos toques detectados), el dispositivo vibra (feedback háptico).
8. La aplicación elige un dedo ganador al azar.
9. Se ejecuta una animación donde:
   - El círculo del ganador se expande/agranda con una animación fluida.
   - Los demás círculos desaparecen gradualmente (fade out).
   - (Opcional) Se emite un sonido de celebración.
10. El ganador permanece visible en la pantalla durante 2-3 segundos.
11. El usuario toca nuevamente para reiniciar un nuevo sorteo, o retorna al menú.

#### Flujo Alternativo A: Menos de 2 toques

1. Si solo hay un dedo en pantalla cuando se cumple el timeout de 3 segundos.
2. Se muestra un mensaje: "Se requieren al menos 2 jugadores para sortear".
3. El sistema no selecciona ganador y reinicia esperando más toques.

#### Flujo Alternativo B: Nuevo sorteo antes de 3 segundos

1. Un nuevo jugador apoya su dedo en pantalla dentro de los 3 segundos de espera.
2. El temporizador se reinicia.
3. Se agrega el nuevo círculo al mapa de toques.

**Postcondiciones:** Se ha seleccionado un dedo ganador. El usuario puede iniciar un nuevo sorteo o abandonar la sección.

## 7. Restricciones Técnicas

### 7.1 Framework y Stack Tecnológico

- **Framework Principal:** React Native con Expo.
- **Base de código unificada:** Código único que compila a iOS (App Store) y Android (Google Play).
- **Lenguaje:** JavaScript / TypeScript.
- **Alternativa de tipado:** Fuerte tipado recomendado (TypeScript).

### 7.2 Almacenamiento Local

- **AsyncStorage:** Para datos simples y rápido acceso (partidas, configuración).
- **SQLite (alternativa):** Si se requiere consultas más complejas o escalabilidad futura.
- **Sin servidor:** Ningún backend, API REST o servicio en la nube.
- **Sin sincronización en la nube:** Datos locales únicamente.

### 7.3 API y Dependencias

- **Sin conexión a Internet requerida:** La app debe funcionar completamente offline.
- **APIs de Expo a utilizar:**
  - `expo-haptics`: Para vibración háptica (sorteo).
  - `react-native-gesture-handler`: Para multitáctil (sorteo).
  - `react-native-reanimated`: Para animaciones fluidas.
  - `expo-app-loading` (si aplica): Para pre-carga de assets.
  - Alternativas nativas: `UIViewPropertyAnimator` (iOS), `ObjectAnimator` (Android).

### 7.4 Plataformas Soportadas

- **iOS:** Versión mínima 12.0 (o superior según Expo).
- **Android:** API Level 21+ (Android 5.0+).
- **Multitáctil:** Soportado nativamente en ambas plataformas.

### 7.5 Persistencia de Datos

- **Formato de almacenamiento:** JSON en AsyncStorage o tablas SQL en SQLite.
- **Estructura de datos para partida:**
  ```json
  {
    "id": "unique-id",
    "name": "Partida Poker",
    "createdAt": "2026-03-18T10:30:00Z",
    "updatedAt": "2026-03-18T11:45:00Z",
    "players": [
      { "id": "p1", "name": "Juan", "score": 150 },
      { "id": "p2", "name": "María", "score": 200 }
    ]
  }
  ```

### 7.6 Vibración y Feedback Háptico

- **Módulo:** `expo-haptics` o API nativa.
- **Trigger:** Al completar el timeout de 3 segundos en el sorteo.
- **Patrón:** Vibración corta (100-200ms).

### 7.7 Multitáctil (Multi-touch)

- **Detección:** React Native Gesture Handler o Responder System nativo.
- **Eventos:** Captura de `onTouchStart`, `onTouchMove`, `onTouchEnd`.
- **Límite:** Mínimo 10 puntos táctiles simultáneos (capacidad máxima típica de dispositivos).

### 7.8 Animaciones

- **Framework:** React Native Reanimated (recomendado) o Animated API.
- **Framerate:** 60fps o superior (60Hz refresh rate de pantalla).
- **Easing:** Funciones predefinidas para rotación de dados, expansión de círculos, fade-out.

## 8. Restricciones de Diseño Visual y Accesibilidad

### 8.1 Estética General

- **Tema:** Mesa de juego real.
- **Fondo:** Verde paño (tipo mesa de billar/póker).
- **Elementos decorativos:** Dados con efecto 3D, fichas coloridas con aspecto físico.
- **Paleta de colores:** Vivos, contrastados, inspirados en juegos de mesa.

### 8.2 Tipografía

- **Textos funcionales:** Fuente sans-serif legible (ej: Roboto en Android, San Francisco en iOS).
- **Tamaño mínimo:** 16sp (48px lógicos).
- **Puntajes decorativos:** Fuente manuscrita (ej: Comic Sans, Caveat, Indie Flower).
- **Peso:** Regular para cuerpo, Bold para títulos.

### 8.3 Colores y Contraste

- **Contraste mínimo:** WCAG AA (4.5:1 para texto pequeño, 3:1 para texto grande).
- **Paleta de círculos (sorteo):** Rojo, azul, verde, amarillo, naranja, púrpura, rosa, cián.
- **Validación:** Verificar contraste contra fondos.

### 8.4 Interactividad y Botones

- **Tamaño mínimo táctil:** 48x48dp (48x48 puntos lógicos).
- **Padding:** 8-16dp alrededor de botones.
- **Feedback visual:** Cambio de color, elevación (shadow) o escala al tocar.
- **Iconografía:** Clara, acompañada de etiquetas de texto (no solo iconos).

### 8.5 Soporte de Tamaños de Texto

- **iOS:** Dynamic Type (preferredContentSizeCat, scaledFont).
- **Android:** Escalas em (sp units) que responden a preferencias de accesibilidad.
- **Límites:** Soportar desde tamaño pequeño (−2) hasta muy grande (+3).

### 8.6 Modo Oscuro

- **Soporte:** Detectar preferencia del sistema (dark mode).
- **Adaptación:** Colores invertidos o paleta alternativa manteniendo contraste.

## 10. Supuestos y Dependencias

### 10.1 Supuestos Técnicos

1. **Multitáctil disponible:** Se asume que todos los dispositivos iOS y Android soportan multitáctil. (Válido para iOS 5+ y Android 3.0+.)
2. **Almacenamiento local:** Se asume que AsyncStorage (≤100MB) o SQLite es suficiente para el número de partidas esperado.
3. **Vibración háptica:** Se asume disponibilidad de API de vibración en iOS y Android. (Disponible desde iOS 6+ y Android 5.0+.)
4. **Expo como base:** Se asume que Expo proporciona las APIs necesarias sin requerir código nativo personalizado.
5. **Generación de números aleatorios:** La plataforma nativa proporciona suficiente entropía para dados (no criptográfico).
6. **Framerate:** Se asume pantallas de 60Hz como mínimo (estándar en dispositivos modernos).

### 10.2 Supuestos del Usuario

1. **Uso local:** Se asume que todos los jugadores comparten el mismo dispositivo o tablet.
2. **Conexión de internet:** No es requerida; los usuarios pueden jugar sin conexión.
3. **Privacidad:** Se asume que los usuarios aceptan almacenamiento local sin sincronización en la nube.
4. **Acceso inmediato:** Se asume que los usuarios desean acceso sin registro.

### 10.3 Dependencias Externas

1. **Expo CLI y EAS Build:** Para compilación y distribución.
2. **App Store Connect (Apple):** Para publicar en iOS.
3. **Google Play Console (Google):** Para publicar en Android.
4. **Librerías de React Native:** `react-native-gesture-handler`, `react-native-reanimated`, `expo-haptics`, etc.

### 10.4 Dependencias de Desarrollo

1. **Node.js 14+:** Para ejecutar Expo y React Native.
2. **npm o yarn:** Gestor de dependencias.
3. **Git:** Control de versiones.
4. **Emulador de iOS (Xcode) o Android (Android Studio):** Para pruebas locales.

### 10.5 Restricciones Asumidas

1. **Máximo de jugadores:** 8 por partida (limitación UI y UX; puede escalarse en futuro).
2. **Máximo de dados:** 6 simultáneamente (restricción de UI; evita desorden visual).
3. **Máximo de partidas guardadas:** 100 (pueden implementarse estrategias de limpieza; depende de política de almacenamiento).
4. **Tiempo de sorteo:** 3 segundos de estabilidad antes de seleccionar ganador (configurable si es necesario).

## 13. Observaciones y Notas Adicionales

1. **Escalabilidad futura:** La arquitectura debe permitir agregar más módulos (ej: spinner, moneda, temporizador) sin rediseñar la navegación principal.

2. **Configuración de puntaje:** En versiones futuras, considerar permitir incrementos personalizables (+1, +5, +10, etc.) por partida.

3. **Historial de movimientos:** Aunque fuera de alcance en v1, guardar un registro de cambios de puntaje facilitaría auditoría y características futuras.

4. **Sonidos y vibraciones:** Feedback háptico implementado; sonidos pueden agregarse en v1.1 si se desea.

5. **Modo dark:** Recomendado para v1 para mejorar accesibilidad y experiencia nocturna.

6. **Temas visuales:** La temática de mesa de juego es fuerte en v1; personalización de temas puede explorarse post-MVP.

7. **Documentación de usuario:** Se recomienda crear un pequeño tutorial integrado o help screens para primeros usuarios.

8. **Métricas y analytics:** Sin backend, no se capturan analíticas; considerar Sentry (móvil) para errores en futuro.
