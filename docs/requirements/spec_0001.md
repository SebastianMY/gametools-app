# Functional Specification Document
## Game Companion Mobile Application

---

## 1. Project Summary

**Project Name:** Game Companion Mobile Application

**Version:** 1.0

**Date:** March 18, 2026

**Overview:**
Game Companion is a mobile application designed for iOS and Android platforms that serves as a digital replacement for physical tools commonly used in card games and board games. The application provides three core utilities: a dice roller, a score tracker, and a drawing/selection mechanism for determining turn order or winners. The app operates entirely offline with no backend infrastructure, user authentication, or cloud synchronization required. The interface is designed to be intuitive and visually inspired by an authentic game table aesthetic, making it accessible to players of all ages.

---

## 2. Objectives and Scope

### 2.1 Primary Objectives

1. **Replace Physical Game Tools:** Eliminate the need for physical dice, score sheets, and drawing mechanisms by providing digital alternatives accessible from a single mobile device.

2. **Offline-First Experience:** Ensure 100% offline functionality with no internet connection required at any point.

3. **Zero-Friction Onboarding:** Enable users to begin using the application immediately without account creation, registration, or authentication.

4. **Persistent Game State:** Automatically save and restore game sessions without user intervention, allowing gameplay to be interrupted and resumed seamlessly.

5. **Intuitive User Experience:** Design an interface that is visually appealing, requires minimal cognitive load, and accommodates users of varying technical proficiency and age groups.

### 2.2 Scope

**Included in Scope:**

- Dice rolling functionality with customizable quantity (1-6 dice)
- Score tracking for up to 8 players per game session
- Multi-touch drawing mechanism for randomized player selection
- Local data persistence using device storage
- Visual animations and haptic feedback
- Support for multiple concurrent game sessions
- Dynamic text sizing for accessibility
- Compatible display across phones and tablets

**Not Included in Scope (v1):**

- User accounts or authentication
- Cloud synchronization or backup
- Game-specific rule implementations (Truco, UNO, etc.)
- Multiplayer networking functionality
- In-app purchases or advertisement
- Social features (sharing, leaderboards)
- Statistical tracking or analytics
- Custom game rule builders

## 3. System Actors

**Description:** The sole actor in this system. Any individual who opens the application without requiring account credentials or authentication.

**Characteristics:**

## 4. Main Use Cases

### 4.1 Use Case: Roll Dice

**UC-001: Dice Rolling**

**Primary Actor:** Anonymous User

**Preconditions:**
- Application is installed and open
- User has navigated to the Dice section

**Main Flow:**

1. User views the Dice interface
2. User selects the desired number of dice (1 to 6) using a visual control
3. User taps the "Roll" button
4. The dice display an animated spinning effect for 0.5-1.5 seconds
5. The animation concludes and individual die values are displayed
6. The sum total of all dice is calculated and prominently displayed
7. User can tap the roll button again to roll again with the same configuration
8. User can adjust the number of dice and roll again

**Alternative Flows:**

- **AF-001-A: Change Dice Quantity Mid-Session**
  1. User taps or adjusts the dice quantity selector
  2. The new quantity is applied immediately
  3. Previous roll results are cleared
  4. User rolls again with the new configuration

- **AF-001-B: View Previous Roll**
  1. After rolling, the results remain visible on screen
  2. User can scroll or navigate to view historical rolls if supported
  3. User can return to rolling state by tapping Roll again

**Postconditions:**
- Dice results are displayed clearly
- User can perform another roll immediately
- No data persistence is required for dice rolls

**Non-Functional Requirements:**
- Animation runs at 60 FPS
- Spin animation duration: 0.5-1.5 seconds
- Results display updates instantly after animation concludes
- Touch response: < 100ms latency

### 4.2 Use Case: Manage Score Tracking

**UC-002: Create Game Session**

**Primary Actor:** Anonymous User

**Preconditions:**
- Application is open
- User has navigated to the Score section
- No active game session, or user selects "New Game"

**Main Flow:**

1. User taps "New Game" or "Create Game"
2. Interface prompts for player names (up to 8 players)
3. User enters names for each participating player
4. User confirms and creates the session
5. The app initializes each player with a score of 0
6. The game session screen displays all players and their current scores

**Alternative Flows:**

- **AF-002-A: Load Previously Saved Game**
  1. User navigates to Score section
  2. User views list of saved game sessions
  3. User taps on a previous game
  4. The app loads the exact state from the last session (player names, scores, order)
  5. User can resume play or view final results

- **AF-002-B: Insufficient Player Input**
  1. User attempts to create a game with no player names
  2. Interface displays a validation message
  3. User must enter at least 1 player name to proceed
  4. User enters names and creates the session

**Postconditions:**
- Game session is created and ready for scoring
- Session is automatically persisted to local storage
- Score display is initialized and visible

**Non-Functional Requirements:**
- Game creation occurs within 1 second
- Player name input has minimum 48x48dp touch target
- Input field supports up to 20 characters per name

---

**UC-003: Track and Modify Player Scores**

**Primary Actor:** Anonymous User

**Preconditions:**
- An active game session exists and is displayed
- Players and their current scores are visible

**Main Flow:**

1. User identifies the player whose score needs modification
2. User taps the "+" button adjacent to the player's name to add points
3. User taps the "−" button adjacent to the player's name to subtract points
4. The player's score updates immediately on screen
5. The total score change is reflected in the interface
6. The session state is automatically saved to device storage
7. User can modify multiple players' scores in any order
8. User can modify the same player's score multiple times

**Alternative Flows:**

- **AF-003-A: Direct Score Input**
  1. User long-presses or double-taps on a player's score value
  2. An input dialog appears allowing direct numeric entry
  3. User enters a new score value
  4. User confirms the change
  5. The score updates and is persisted

- **AF-003-B: Undo Last Change**
  1. User taps an undo button or performs a gesture
  2. The most recent score modification is reversed
  3. Undo state is limited to recent changes (implementation-specific)

- **AF-003-C: Reset All Scores**
  1. User accesses game settings or menu
  2. User selects "Reset Scores" or "Restart Game"
  3. A confirmation dialog appears
  4. User confirms the reset
  5. All player scores are set to 0
  6. The change is persisted

**Postconditions:**
- Player score is updated accurately
- Updated state is persisted to local storage
- Score change is visible to user immediately

**Non-Functional Requirements:**
- Score modification updates display within 50ms
- Persistent save operation completes within 200ms
- Minimum touch target for +/− buttons: 48x48dp
- Undo operation reverses to exact previous state

---

**UC-004: Persist and Resume Game Session**

**Primary Actor:** Anonymous User

**Preconditions:**
- An active game session exists
- The app is running or the user may close the app

**Main Flow:**

1. During active gameplay, every score modification triggers an automatic save
2. The app saves the complete game state: player names, scores, timestamps
3. User can close the app at any time
4. User reopens the app on the same device (hours or days later)
5. The app detects previously saved games and displays them
6. User selects a saved game from the list
7. The app restores the exact state: player names, scores, all modifications
8. Gameplay can resume immediately

**Alternative Flows:**

- **AF-004-A: Delete Saved Game**
  1. User opens the Score section and views saved games
  2. User performs a delete action on a saved game (swipe, long-press, menu)
  3. A confirmation dialog appears
  4. User confirms deletion
  5. The game session is permanently removed from device storage

- **AF-004-B: Multiple Concurrent Games**
  1. During an active game session, user creates a new game
  2. The previous session is automatically saved
  3. The new game becomes the active session
  4. User can switch between saved games by accessing the game list
  5. Each game maintains its independent state

**Postconditions:**
- Game state is persisted to device storage
- Session can be resumed from any point where it was interrupted
- Multiple games are stored and retrievable

**Non-Functional Requirements:**
- Persistence operation completes in < 200ms
- Data integrity is maintained across app restarts
- No data loss on forced app termination
- Storage capacity supports at least 50 concurrent game sessions

### 4.3 Use Case: Execute Multi-Touch Drawing

**UC-005: Perform Random Selection via Multi-Touch Drawing**

**Primary Actor:** Anonymous User (typically used by 2+ users on same device)

**Preconditions:**
- Application is open
- User has navigated to the Draw section
- Multiple participants (players) are present and ready

**Main Flow:**

1. User views the Draw interface with instructions
2. All participating players simultaneously place one finger on the device screen
3. Each touch point is detected and assigned a unique color (red, blue, yellow, green, purple, orange, pink, teal, etc.)
4. Each touch point is represented by a colored circle that expands or pulses to indicate active contact
5. The app monitors touch stability for 3 seconds
6. After 3 seconds of stable touches, the device vibrates (haptic feedback)
7. The app randomly selects one of the active touch points as the winner
8. The winner's circle enlarges with an animation and is highlighted prominently
9. All other circles fade out or disappear over 0.5-1 second
10. The winning circle remains visible for 2-3 seconds before returning to the initial state

**Alternative Flows:**

- **AF-005-A: Touch Release During Wait**
  1. One or more players release their finger before the 3-second timer completes
  2. The corresponding colored circle fades out immediately
  3. If at least 1 touch remains, the timer continues
  4. If all touches are released, the timer resets to 0
  5. Players can reapply touches to restart the process

- **AF-005-B: Unequal Number of Participants**
  1. Fewer or more than expected participants place their fingers
  2. The app detects all active touches (no minimum or maximum enforced)
  3. The selection occurs among all detected touches
  4. The process completes normally

- **AF-005-C: Immediate Selection**
  1. User can tap a "Select Now" button to force selection before 3 seconds elapse
  2. The app vibrates and randomly selects from current touches
  3. Results display as normal

**Postconditions:**
- A winning touch point is clearly identified
- The result is visible for sufficient time to be observed
- Interface returns to ready state for another draw

**Non-Functional Requirements:**
- Multi-touch detection supports minimum 10 simultaneous touch points
- Touch response latency: < 50ms
- Color differentiation: sufficient contrast for visibility (WCAG AA minimum 4.5:1)
- Vibration duration: 50-200ms
- Animation frame rate: 60 FPS
- 3-second timer is accurate to within 100ms

## 5. Functional Requirements

### 5.1 Dice Feature

**FR-D-001:** The application shall allow users to select a quantity of dice between 1 and 6 (inclusive).

**FR-D-002:** The application shall provide a visual control for dice quantity selection (spinner, slider, or segmented control).

**FR-D-003:** Upon user action, the application shall animate all dice with a spinning effect for a duration of 0.5–1.5 seconds.

**FR-D-004:** After animation concludes, the application shall display the numeric value (1–6) of each individual die.

**FR-D-005:** The application shall calculate and display the sum total of all rolled dice prominently.

**FR-D-006:** The application shall allow users to roll again immediately with the same or modified dice quantity.

**FR-D-007:** The application shall not require persistence of dice roll results; rolls are ephemeral.

### 5.2 Score Tracking Feature

**FR-S-001:** The application shall allow creation of a new game session with up to 8 players.

**FR-S-002:** During game session creation, the user shall input names for participating players.

**FR-S-003:** The application shall initialize all players with a score of 0 at session creation.

**FR-S-004:** The application shall provide +/− controls (buttons) for each player to modify their score.

**FR-S-005:** Each +/− action shall modify the player's score by 1 point (adjustable to other increments as desired).

**FR-S-006:** The application shall display scores persistently and update immediately upon modification.

**FR-S-007:** The application shall automatically persist the complete game state (player names, scores) to local device storage after every score change.

**FR-S-008:** The application shall retrieve and restore saved game sessions when the user navigates to the Score section.

**FR-S-009:** The application shall support multiple concurrent saved game sessions on a single device.

**FR-S-010:** The application shall allow users to delete saved game sessions.

**FR-S-011:** The application shall display a list of all saved games with identifiable information (player names, timestamp, etc.).

**FR-S-012:** The application shall allow users to resume a previously saved game from the exact state in which it was interrupted.

### 5.3 Multi-Touch Drawing (Draw) Feature

**FR-T-001:** The application shall detect and track up to 10 simultaneous touch points on the device screen.

**FR-T-002:** Each detected touch point shall be represented by a colored circle on screen.

**FR-T-003:** The application shall assign a unique color to each touch point from a predefined palette.

**FR-T-004:** The application shall monitor touch stability; when all touches remain stationary for 3 seconds, the selection mechanism shall activate.

**FR-T-005:** Upon 3 seconds of touch stability, the device shall produce haptic feedback (vibration).

**FR-T-006:** The application shall randomly select one of the active touch points as the winner with equal probability.

**FR-T-007:** The winner's circle shall enlarge with an animation to visually distinguish it from other circles.

**FR-T-008:** Non-winning circles shall fade or disappear over 0.5–1 second.

**FR-T-009:** The winner's circle shall remain visible for 2–3 seconds before returning to the initial ready state.

**FR-T-010:** If a touch is released before the 3-second timer completes, the corresponding circle shall disappear and the timer shall reset or continue based on remaining touches.

**FR-T-011:** Users shall be able to force immediate selection (before 3 seconds) via an on-screen button.

**FR-T-012:** The application shall display clear instructions on the Draw interface explaining multi-touch requirements.

## 6. Non-Functional Requirements

### 6.1 Performance

**NFR-P-001:** All animations (dice rolling, circle enlargement, fade effects) shall run at 60 frames per second (FPS) on target devices.

**NFR-P-002:** Touch input response latency shall not exceed 100 milliseconds.

**NFR-P-003:** Score modification shall update the display within 50 milliseconds of user action.

**NFR-P-004:** Persistent save operations shall complete within 200 milliseconds.

**NFR-P-005:** Game session restoration from storage shall complete within 500 milliseconds.

**NFR-P-006:** The application shall support smooth performance on devices with minimum 2GB RAM.

### 6.2 Reliability and Data Integrity

**NFR-R-001:** Game session state shall be persisted after every user action that modifies score.

**NFR-R-002:** No data loss shall occur if the application is forcibly terminated or the device loses power during a game session.

**NFR-R-003:** The application shall maintain data integrity across app updates.

**NFR-R-004:** Multiple concurrent saved game sessions shall be stored independently without interference.

### 6.3 Offline Operation

**NFR-O-001:** The application shall function 100% offline with no internet connection required.

**NFR-O-002:** The application shall not attempt to establish network connections or fail gracefully if connection is attempted and unavailable.

**NFR-O-003:** All data shall be stored and retrieved exclusively from local device storage.

### 6.4 Compatibility

**NFR-C-001:** The application shall run on iOS 13.0 or later.

**NFR-C-002:** The application shall run on Android 8.0 (API level 26) or later.

**NFR-C-003:** The application shall support a variety of screen sizes: phones (5–7 inches) and tablets (7–12 inches).

**NFR-C-004:** The application shall maintain a single shared codebase for iOS and Android (React Native requirement).

### 6.5 Accessibility

**NFR-A-001:** All interactive elements (buttons, touch targets) shall have a minimum size of 48×48 device-independent pixels (dp).

**NFR-A-002:** Text contrast ratios shall meet WCAG 2.1 AA standard (minimum 4.5:1 for body text, 3:1 for large text).

**NFR-A-003:** The application shall support system-level text sizing (iOS Dynamic Type, Android Text scaling).

**NFR-A-004:** All interactive elements shall be clearly labeled with descriptive text or icons.

**NFR-A-005:** The application shall support VoiceOver (iOS) and TalkBack (Android) screen readers for critical interface elements.

**NFR-A-006:** Color shall not be the sole indicator of information; text labels and icons shall accompany all colors.

### 6.6 User Experience

**NFR-U-001:** The application shall provide visual and haptic feedback for all user actions.

**NFR-U-002:** The interface shall use intuitive visual metaphors (game table aesthetic) that do not require instruction.

**NFR-U-003:** Onboarding shall occur seamlessly with no login or registration required.

**NFR-U-004:** Navigation between sections (Dice, Score, Draw) shall be straightforward and require minimal user cognitive load.

## 7. Technical Constraints

### 7.1 Technology Stack

**Framework:** React Native with Expo

**Rationale:** Single codebase supporting both iOS and Android platforms, rapid development, and simplified build/deployment process.

**Language:** JavaScript / TypeScript (recommended)

**Local Storage:** AsyncStorage or SQLite

- **AsyncStorage:** Simple key-value store suitable for game session data (recommended for v1 simplicity)
- **SQLite:** More robust option if structured queries are needed in future iterations

**Vibration API:** Expo Haptics module

**Multi-touch Handling:** React Native PanResponder or GestureHandler library

**Animation:** React Native Animated API or Reanimated library for complex animations

### 7.2 Infrastructure

**No Backend:** This application has no server-side infrastructure or API layer.

**No Cloud Services:** No cloud storage, synchronization, or backup services.

**Device-Local Storage:** All data persists exclusively on the user's device using native storage APIs.

### 7.3 Authentication and Security

**No Authentication:** The application has no user accounts, login mechanism, or authentication layer.

**No Authorization:** All users have equal access to all features.

**Device Security:** The application respects the device's native security model (iOS Keychain for sensitive data if needed in future versions, Android KeyStore).

**Data Encryption:** Local data may optionally be encrypted at rest using device-native encryption (Android Encrypted SharedPreferences, iOS Secure Enclave) for future versions; not required for v1.

### 7.4 Platform-Specific Considerations

**iOS (Apple App Store):**
- Minimum deployment target: iOS 13.0
- Requires App Store review and approval
- Must comply with App Store guidelines (no account creation requirement confirmed)

**Android (Google Play Store):**
- Minimum API level: 26 (Android 8.0)
- Requires Google Play review and approval
- Must comply with Google Play policies

**Build and Deployment:**
- Expo EAS Build for managed CI/CD
- Expo EAS Submit for automated app store submission
- No custom native modules required for v1

## 9. Assumptions and Dependencies

### 9.1 Assumptions

**A-001:** Users will possess a modern smartphone or tablet running iOS 13+ or Android 8.0+ and will maintain these devices during gameplay.

**A-002:** Users are familiar with basic mobile device interactions (tapping, scrolling, multi-touch gestures) and do not require extensive onboarding.

**A-003:** Users will use the application in a casual, social setting with others present (especially for the Draw feature), but the app itself runs on a single physical device.

**A-004:** Game sessions are typically short (minutes to hours) and do not require long-term historical tracking across multiple days.

**A-005:** Users do not require advanced statistical analysis or detailed historical records of past games.

**A-006:** Device storage capacity is sufficient for multiple game sessions (conservative estimate: < 5MB per 50 sessions).

**A-007:** Users understand that without internet connection, there is no cloud backup or recovery if the device is reset or lost.

**A-008:** The application is intended for casual gaming environments and does not require compliance with regulated industries (gambling, financial, healthcare).

**A-009:** Haptic feedback (vibration) is available on all target devices; graceful degradation occurs if unavailable.

**A-010:** Multi-touch is supported on all iOS and Android devices within the target range; the Draw feature functions as designed.

### 9.2 Dependencies

**D-001:** Expo framework and toolchain (maintained by Expo Inc.) for consistent iOS/Android build management.

**D-002:** React Native framework (open source, maintained by Meta and community) for cross-platform mobile development.

**D-003:** Apple App Store and Google Play Store infrastructure for application distribution.

**D-004:** Device manufacturers' provision of multi-touch input, haptic feedback, and local storage APIs.

**D-005:** Operating system support for Expo-managed native code (iOS and Android native libraries for Haptics, Storage, etc.).

**D-006:** TypeScript/JavaScript toolchain and dependency managers (npm or yarn) for development.

**D-007:** Version control system (Git) and repository hosting (GitHub, GitLab, Bitbucket, etc.) for source code management.

### 9.3 Known Constraints and Mitigations

**Constraint C-001:** Expo-managed projects have limitations on custom native module integration.

- **Mitigation:** v1.0 requirements can be fully met with Expo's built-in modules (Haptics, AsyncStorage, etc.). If custom native code is needed in future versions, EAS can be configured to support expo-plugins or migration to bare React Native.

**Constraint C-002:** AsyncStorage is not recommended for large datasets or frequent writes in production.

- **Mitigation:** For v1.0, AsyncStorage is sufficient for the expected data volume (few dozen game sessions with modest structure). If performance degrades, migration to SQLite is straightforward.

**Constraint C-003:** React Native performance on lower-end Android devices may be suboptimal for 60 FPS animations.

- **Mitigation:** Target devices with minimum 2GB RAM; optimize animations using React Native's Animated API or Reanimated; conduct testing on actual devices to measure and optimize performance.

**Constraint C-004:** Multi-touch detection may conflict with other gesture handlers if not carefully managed.

- **Mitigation:** Use a dedicated, isolated gesture handler for the Draw feature; ensure no conflicting responders are active during Draw.

## 10. Appendix: Visual and Interaction Guidelines

### 10.1 Design Aesthetic

**Theme:** Game Table Aesthetic

The visual design should evoke the experience of playing games around a physical table:
- Background: Green felt or fabric texture reminiscent of a gaming table
- Dice: Rendered with subtle 3D perspective and shadows to mimic physical dice
- Score displays: Manuscript or handwritten-style typography for a natural, organic feel
- Colors: Vibrant, distinct colors for player differentiation (inspired by game tokens/chips)
- Spacing: Generous padding and clear visual hierarchy

### 10.2 Color Palette (Recommended)

**Primary Actions:** Accent color (e.g., golden yellow or bright blue) for buttons and interactive elements

**Multi-Touch Circle Colors:**
1. Red (#FF6B6B)
2. Blue (#4ECDC4)
3. Yellow (#FFE66D)
4. Green (#95E77D)
5. Purple (#C7B3E5)
6. Orange (#FFAB91)
7. Pink (#FF85B3)
8. Teal (#7DD3C0)

**Text:** Dark gray or black on light backgrounds; light gray or white on dark backgrounds

**Contrast:** Maintain minimum 4.5:1 ratio for all body text (WCAG AA)

### 10.3 Typography

**Functional Text:** Clear sans-serif font (e.g., San Francisco, Roboto) at minimum 16sp

**Score/Decorative:** Optional manuscript/hand-drawn style font for visual appeal (secondary usage)

**Dynamic Sizing:** Respect system font scaling on iOS (Dynamic Type) and Android (Text scaling)

### 10.4 Button and Touch Target Sizing

**Minimum Size:** 48×48 dp (44×44 pt on iOS, 48×48 dp on Android) for all interactive elements

**Recommended Spacing:** 8-16 dp margin between adjacent touch targets to prevent accidental interaction

### 10.5 Animations

**Dice Roll:** Spinning/rotating animation, 0.5–1.5 seconds, easing function (e.g., easeOut)

**Winner Circle:** Scale and fade-in animation, 0.3–0.5 seconds

**Loser Circles:** Fade-out and scale-down animation, 0.5–1 second

**All animations:** 60 FPS frame rate
